# LLM Wiki

一種利用 LLM 建構個人知識庫的模式。

這是一份概念文件，設計上可以直接複製貼到你自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目標是傳達高層次的概念，而你的 Agent 會在與你的協作中建構出具體的細節。

## 核心概念

大多數人使用 LLM 處理文件的經驗看起來像 RAG：你上傳一批檔案，LLM 在查詢時檢索相關片段，然後生成回答。這行得通，但 LLM 每次都是從零開始重新發現知識。沒有任何累積。問一個需要綜合五份文件的細微問題，LLM 每次都得找到並拼湊相關片段。什麼都沒有被建立起來。NotebookLM、ChatGPT 檔案上傳，以及大多數 RAG 系統都是這樣運作的。

這裡的概念截然不同。不只是在查詢時從原始文件中檢索，LLM 會**漸進式地建構並維護一個持久的 Wiki**——一個結構化、相互連結的 markdown 檔案集合，介於你和原始資料來源之間。當你新增一個來源時，LLM 不只是為了之後檢索而索引它。它會閱讀它、萃取關鍵資訊，並整合到現有的 Wiki 中——更新實體頁面、修訂主題摘要、標記新資料與舊主張的矛盾之處、強化或挑戰不斷演進的綜合分析。知識只需編譯一次，然後*持續更新*，而非每次查詢都重新推導。

這就是關鍵差異：**Wiki 是一個持久的、不斷累積的產物。** 交叉引用已經建立好了。矛盾已經被標記出來了。綜合分析已經反映了你讀過的所有內容。每新增一個來源、每問一個問題，Wiki 都會變得更加豐富。

你永遠不需要（或很少需要）親自撰寫 Wiki——LLM 會撰寫並維護所有內容。你負責資料來源的策展、探索，以及問對的問題。LLM 負責所有繁瑣的工作——摘要、交叉引用、歸檔，以及讓知識庫隨時間真正發揮價值的簿記工作。在實務上，我一邊開著 LLM Agent，另一邊開著 Obsidian。LLM 根據我們的對話進行編輯，我即時瀏覽結果——追蹤連結、查看圖譜視圖、閱讀更新的頁面。Obsidian 是 IDE；LLM 是程式設計師；Wiki 是程式碼庫。

這可以應用於許多不同的情境。以下是一些例子：

- **個人**：追蹤你自己的目標、健康、心理、自我提升——歸檔日記、文章、播客筆記，隨著時間建構一幅關於自己的結構化圖像。
- **研究**：在數週或數月內深入研究一個主題——閱讀論文、文章、報告，漸進式地建構一個包含不斷演進論點的全面性 Wiki。
- **閱讀書籍**：逐章歸檔，為角色、主題、情節線索建立頁面，以及它們之間的關聯。讀完後你會有一個豐富的伴讀 Wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 這樣的粉絲 Wiki——數千個相互連結的頁面涵蓋角色、地點、事件、語言，由志工社群歷經數年建構。你可以在閱讀時個人建構類似的東西，由 LLM 負責所有的交叉引用和維護。
- **商業/團隊**：由 LLM 維護的內部 Wiki，接收 Slack 對話串、會議逐字稿、專案文件、客戶通話記錄。可能由人工在迴圈中審查更新。Wiki 保持最新狀態，因為 LLM 負責團隊中沒人想做的維護工作。
- **競品分析、盡職調查、旅行規劃、課程筆記、興趣深入研究**——任何你隨時間累積知識且希望有組織而非散落各處的場景。

## 架構

有三個層次：

**原始來源** — 你精心策展的來源文件集合。文章、論文、圖片、資料檔。這些是不可變的——LLM 從中讀取但永遠不會修改。這是你的事實來源。

**Wiki** — 一個由 LLM 生成的 markdown 檔案目錄。摘要、實體頁面、概念頁面、比較分析、總覽、綜合分析。LLM 完全擁有這一層。它創建頁面、在新來源到達時更新頁面、維護交叉引用，並保持一切的一致性。你閱讀它；LLM 撰寫它。

**Schema** — 一份文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告訴 LLM Wiki 的結構、慣例是什麼，以及在匯入來源、回答問題或維護 Wiki 時應遵循什麼工作流程。這是關鍵的設定檔——它讓 LLM 成為一個有紀律的 Wiki 維護者，而不是一個通用聊天機器人。你和 LLM 會隨著時間共同演進這份文件，找出什麼最適合你的領域。

## 操作

**匯入（Ingest）。** 你把新來源放入原始資料集合並告訴 LLM 處理它。一個範例流程：LLM 閱讀來源、與你討論關鍵要點、在 Wiki 中撰寫摘要頁面、更新索引、更新 Wiki 中相關的實體和概念頁面，並在日誌中新增一筆記錄。單一來源可能觸及 10-15 個 Wiki 頁面。就我個人而言，我偏好逐一匯入來源並保持參與——我閱讀摘要、檢查更新，並引導 LLM 強調什麼。但你也可以在較少監督下批次匯入多個來源。由你來發展適合你風格的工作流程，並記錄在 Schema 中供未來的對話使用。

**查詢（Query）。** 你針對 Wiki 提問。LLM 搜尋相關頁面、閱讀它們，並綜合出附有引用的回答。回答可以根據問題採取不同形式——markdown 頁面、比較表、投影片（Marp）、圖表（matplotlib）、畫布。重要的洞察：**好的回答可以作為新頁面歸檔回 Wiki。** 你提出的比較、分析、你發現的關聯——這些都很有價值，不應該消失在聊天歷史中。這樣你的探索就像匯入的來源一樣在知識庫中不斷累積。

**檢查（Lint）。** 定期請 LLM 對 Wiki 進行健康檢查。檢查：頁面之間的矛盾、較新來源已取代的過時主張、沒有入站連結的孤立頁面、被提及但缺乏自己頁面的重要概念、缺失的交叉引用、可以透過網路搜尋填補的資料缺口。LLM 擅長建議需要調查的新問題和需要尋找的新來源。這讓 Wiki 隨著成長保持健康。

## 索引與日誌

兩個特殊檔案幫助 LLM（和你）在 Wiki 成長時導航。它們服務於不同的目的：

**index.md** 以內容為導向。它是 Wiki 中所有內容的目錄——每個頁面列出連結、一行摘要，以及可選的後設資料如日期或來源數量。按類別組織（實體、概念、來源等）。LLM 在每次匯入時更新它。回答查詢時，LLM 先讀索引找到相關頁面，再深入閱讀。這在中等規模（約 100 個來源、數百個頁面）下效果出奇地好，避免了需要基於嵌入的 RAG 基礎設施。

**log.md** 以時間順序排列。它是一份只能追加的記錄，記載什麼事在什麼時候發生——匯入、查詢、檢查。一個實用技巧：如果每個條目以一致的前綴開頭（例如 `## [2026-04-02] ingest | 文章標題`），日誌就可以用簡單的 unix 工具解析——`grep "^## \[" log.md | tail -5` 就能看到最後 5 筆記錄。日誌給你 Wiki 演進的時間線，幫助 LLM 了解最近做了什麼。

## 可選：CLI 工具

到了某個階段你可能想建構小工具來幫助 LLM 更有效率地操作 Wiki。Wiki 頁面的搜尋引擎是最明顯的一個——小規模時索引檔就夠了，但隨著 Wiki 成長你需要正式的搜尋。[qmd](https://github.com/tobi/qmd) 是個好選擇：它是一個本地 markdown 檔案搜尋引擎，具備混合 BM25/向量搜尋和 LLM 重排序，全部在本地裝置上運行。它同時具備 CLI（讓 LLM 可以執行 shell 命令呼叫它）和 MCP 伺服器（讓 LLM 可以作為原生工具使用它）。你也可以自己建構更簡單的東西——LLM 可以在需要時幫你氛圍編程一個簡易搜尋腳本。

## 提示與技巧

- **Obsidian Web Clipper** 是一個瀏覽器擴充功能，可將網頁文章轉換為 markdown。非常適合快速將來源收進你的原始資料集合。
- **在本地下載圖片。** 在 Obsidian 設定 → 檔案與連結中，將「附件資料夾路徑」設為固定目錄（例如 `raw/assets/`）。然後在設定 → 快捷鍵中，搜尋「Download」找到「Download attachments for current file」並綁定快捷鍵（例如 Ctrl+Shift+D）。剪輯文章後，按下快捷鍵，所有圖片就會下載到本地磁碟。這是可選的但很實用——它讓 LLM 可以直接查看和引用圖片，而不是依賴可能失效的 URL。注意 LLM 無法在一次操作中原生讀取帶有內嵌圖片的 markdown——變通方法是讓 LLM 先讀取文字，然後分別查看部分或全部引用的圖片以獲得額外脈絡。這有點笨拙但效果不錯。
- **Obsidian 的圖譜視圖** 是查看 Wiki 全貌的最佳方式——什麼與什麼相連、哪些頁面是樞紐、哪些是孤立的。
- **Marp** 是一種基於 markdown 的投影片格式。Obsidian 有相應的外掛。適合直接從 Wiki 內容生成簡報。
- **Dataview** 是 Obsidian 的外掛，可對頁面前置資料執行查詢。如果你的 LLM 在 Wiki 頁面加入 YAML 前置資料（標籤、日期、來源數量），Dataview 可以生成動態表格和列表。
- Wiki 就是一個由 markdown 檔案組成的 git 儲存庫。你免費獲得版本歷史、分支和協作功能。

## 為什麼這行得通

維護知識庫中繁瑣的部分不是閱讀或思考——而是簿記工作。更新交叉引用、保持摘要最新、標記新資料與舊主張的矛盾、在數十個頁面間維持一致性。人類會放棄 Wiki，因為維護負擔的成長速度快過它的價值。LLM 不會厭倦、不會忘記更新交叉引用，而且可以一次操作觸及 15 個檔案。Wiki 能持續被維護，因為維護成本趨近於零。

人類的工作是策展來源、指導分析、問好的問題，以及思考這一切意味著什麼。LLM 的工作是其他所有事情。

這個概念在精神上與 Vannevar Bush 的 Memex（1945）有關——一個個人的、精心策展的知識儲存庫，文件之間有聯想式的路徑。Bush 的願景比後來的網路更接近這個概念：私人的、主動策展的，文件之間的連結與文件本身同等重要。他無法解決的部分是誰來做維護。LLM 處理了這個問題。

## 備註

這份文件刻意保持抽象。它描述的是概念，而非特定的實作。確切的目錄結構、Schema 慣例、頁面格式、工具——所有這些都取決於你的領域、你的偏好，以及你選擇的 LLM。上述提到的一切都是可選且模組化的——選擇有用的，忽略不需要的。例如：你的來源可能只有文字，所以你完全不需要圖片處理。你的 Wiki 可能小到索引檔就夠用了，不需要搜尋引擎。你可能不在乎投影片只想要 markdown 頁面。你可能想要完全不同的輸出格式。正確的使用方式是把這份文件分享給你的 LLM Agent，一起合作實例化一個符合你需求的版本。這份文件唯一的工作就是傳達這個模式。你的 LLM 可以搞定其餘的。
