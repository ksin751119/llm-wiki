第二章 — 架構全景

三層架構

每層有明確的擁有者和不變性。LLM 不是通用聊天機器人——它是有紀律的維護者。

使用者互動層
Ingest / Query / Lint
三大操作定義所有行為
擁有者：人類
定義行為 ↓
Schema 層
CLAUDE.md / AGENTS.md
結構約定、格式規範、工作流定義
擁有者：人 + LLM 共同
遵循規則 ↓
Wiki 層
實體頁 / 概念頁 / 摘要頁 / 比較頁
LLM 寫入，人類閱讀
擁有者：LLM
讀取（不修改）↓
Raw Sources 層
文章、論文、圖片、資料檔
不可變真相，LLM 只讀不寫
擁有者：人類策展

職責分離

每層擁有者明確。Raw Sources 永不被 LLM 修改，保持真相。Wiki 層只被 LLM 寫入，人只閱讀。

六種頁面類型

摘要頁單一來源的重點摘錄
實體頁人物／組織／概念的持久頁面
概念頁跨來源的主題彙整
比較頁兩個以上項目的結構化對比
綜合頁高層次的洞察與趨勢
總覽頁wiki 的整體摘要

Schema 驅動

Schema 讓 LLM 成為有紀律的維護者，而非通用聊天機器人。它定義頁面類型、命名慣例、工作流步驟，隨使用不斷演化。

index.md
空間維度
「wiki 裡有什麼」— 按類別分組的內容目錄。查詢時的第一站，每頁一行：連結 + 摘要。
log.md
時間維度
「wiki 怎麼變的」— Append-only 時間線。追蹤每次操作觸及的頁面，避免重複工作。
第四章 — 核心流程

Ingest：知識的編譯

Ingest 不只是「存檔」。單一來源可能觸發 10–15 個 wiki 頁面的更新。

1
加入來源
將文件放入 raw sources 目錄。可用 Web Clipper 一鍵擷取。
2
LLM 讀取
閱讀全文，提取關鍵資訊、主張、數據。
3
討論重點
使用者引導強調什麼、忽略什麼。判斷讓 wiki 更有價值。
4
級聯更新
建立摘要頁、更新實體頁、概念頁、標記矛盾、加入交叉引用。
5
更新索引
在 index.md 加入新條目，更新已修改頁面的摘要。
6
記錄日誌
在 log.md 追加記錄，列出所有被觸及的頁面。
級聯更新的一次觸發
  • 建立該來源的摘要頁
  • 掃描既有 wiki，找到所有相關頁面
  • 更新實體頁（新資訊、修正舊資訊）
  • 更新概念頁（加入新觀點或證據）
  • 標記任何矛盾（新來源 vs 舊來源）
  • 加入交叉引用連結
第五章 — 進階流程

Query 與 Lint

知識不只是存入，還要能取出、能自我修復。

Query — 知識提取
問答產生新知識
讀取 index.md 確定相關頁面範圍
讀取 3–10 頁相關 wiki 頁面
綜合答案，附引用來源頁面
值得保留？存為新 wiki 頁面
更新 index.md + log.md
Lint — 健康檢查
wiki 的自我修復
偵測矛盾：新來源推翻舊主張？
過時聲明：已被更新資料取代？
孤兒頁面：無入站連結的頁面？
缺失引用：應有交叉連結但沒有？
建議下一步該研究什麼
一次比較分析、一個深度問答、一個發現的連結——這些不該消失在聊天歷史中。回填到 wiki 後，它們成為未來查詢可引用的持久知識。

第六章 — 關鍵設計決策

為什麼不是 RAG？

知識編譯與即時檢索，根本性的差異。

維度 RAG LLM Wiki
知識狀態每次從零開始持久累積
交叉引用查詢時才拼湊已預先建立
矛盾處理不處理已標記並追蹤
綜合深度受檢索品質限制反映全部已讀內容
維護成本無（也無累積）LLM 負擔，接近零

維護知識庫的繁瑣部分不是閱讀或思考——是記帳。人類放棄 wiki 是因為維護成本成長快於價值。LLM 不會厭倦。

— Vannevar Bush 的 Memex 願景，由 LLM 實現

第八章 — 實踐指南

如何開始

1

選擇領域和工具

決定主題（研究、讀書、個人成長），選擇 LLM agent 和瀏覽器。

2

建立基本結構

建立 raw/ 和 wiki/ 目錄，初始化 index.md 和 log.md，加入 git 版控。

3

撰寫 Schema

與 LLM 共同建立初版 CLAUDE.md，定義頁面類型、命名慣例、工作流步驟。

4

開始 Ingest

從一個來源開始，逐步建立。觀察 LLM 如何處理，根據需要調整 Schema。

5

定期 Lint

每週或每 10 個來源後，請 LLM 做健康檢查。修復問題，建議下一步研究方向。

你是策展人，不是作者。
你選擇來源和方向，LLM 做記帳。

一切都是可選的 — 從最小可行 wiki 開始
