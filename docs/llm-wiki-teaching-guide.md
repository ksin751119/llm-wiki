## 第一章：總覽與定位

### LLM Wiki 解決什麼問題？

大多數人使用 LLM 處理文件的方式是 **RAG（Retrieval-Augmented Generation）**：上傳文件集合，LLM 在查詢時檢索相關片段再生成答案。問題在於——LLM 每次都從零開始重新發現知識，**沒有累積**。一個需要綜合五篇文件的微妙問題，RAG 每次都得重新拼湊。NotebookLM、ChatGPT 檔案上傳等工具都是這種模式。

LLM Wiki 提出不同思路：LLM **增量建構並維護一個持久化 wiki** — 一個結構化、互相連結的 markdown 檔案集合。每當加入新來源，LLM 不只是索引它，而是讀取、提取關鍵資訊、**整合到現有 wiki** — 更新實體頁面、修訂主題摘要、標記矛盾、強化或挑戰既有綜合。

### 一句話摘要

**Wiki 是持久化的編譯產物，不是即時的檢索結果。** 知識編譯一次，持續更新，查詢時只需讀取已編譯的成果。

### 核心角色分工

| 角色 | 職責 |
|------|------|
| **人類** | 策展來源、引導分析、提出好問題、思考意義 |
| **LLM** | 摘要、交叉引用、歸檔、記帳 — 所有繁瑣的維護工作 |

### 適用場景

- **個人成長**：追蹤目標、健康、心理、自我改善
- **研究**：數週/月的深度研究，論文、報告的增量綜合
- **讀書**：逐章建立角色、主題、情節線 wiki（如 Tolkien Gateway）
- **團隊/商業**：由 LLM 維護的內部 wiki，來源為 Slack、會議紀錄、專案文件
- **其他**：競爭分析、盡職調查、旅行計畫、課程筆記

### 前置知識需求

- 基本 markdown 語法
- 了解檔案系統與目錄結構
- （可選）Obsidian 使用經驗
- （可選）基本 git 操作

## 第二章：架構全景

### 三層架構

LLM Wiki 由三個明確分離的層構成，每層有不同的擁有者和不變性：

```
┌─────────────────────────────────────────────┐
│           使用者互動層                        │
│     Ingest / Query / Lint 三大操作           │
└──────────────┬──────────────────────────────┘
               │ 定義行為
┌──────────────▼──────────────────────────────┐
│   Schema 層 (CLAUDE.md / AGENTS.md)          │
│   • wiki 結構約定    • 頁面格式規範          │
│   • 工作流定義        • 命名慣例             │
│   擁有者：人 + LLM 共同演化                  │
└──────────────┬──────────────────────────────┘
               │ 遵循 Schema 規則
┌──────────────▼──────────────────────────────┐
│   Wiki 層 (LLM 完全擁有)                     │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐      │
│   │ 實體頁  │ │ 概念頁  │ │ 比較頁  │      │
│   └─────────┘ └─────────┘ └─────────┘      │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐      │
│   │ 摘要頁  │ │ 綜合頁  │ │ 總覽頁  │      │
│   └─────────┘ └─────────┘ └─────────┘      │
│   ┌───────────────┐ ┌──────────────┐        │
│   │ index.md 目錄 │ │ log.md 日誌  │        │
│   └───────────────┘ └──────────────┘        │
│   擁有者：LLM（人閱讀，LLM 寫入）           │
└──────────────┬──────────────────────────────┘
               │ 讀取（不修改）
┌──────────────▼──────────────────────────────┐
│   Raw Sources 層 (不可變真相)                │
│   文章、論文、圖片、資料檔                   │
│   擁有者：人類策展                           │
│   不變性：LLM 只讀不寫                       │
└─────────────────────────────────────────────┘
```

### 資料流向圖

```
新來源 ──Ingest──> LLM 讀取 ──> 產生/更新 wiki 頁面
                                    │
                              更新 index.md
                              追加 log.md
                                    │
                                    ▼
問題 ──Query──> 讀 index ──> 找相關頁面 ──> 綜合答案
                                              │
                                    [可選] 回填為新 wiki 頁面
                                              │
                                              ▼
         ──Lint──> 掃描 wiki ──> 標記矛盾/孤兒/缺口
                                    │
                              修復問題頁面
                              建議新來源/問題
```

### 關鍵設計原則

1. **職責分離**：每層擁有者明確（人/LLM/共同）
2. **不可變來源**：Raw Sources 永不被 LLM 修改，保持真相
3. **單向寫入**：wiki 層只被 LLM 寫入，人只閱讀
4. **Schema 驅動**：Schema 讓 LLM 成為有紀律的維護者，而非通用聊天機器人

## 第三章：核心資料結構

### Wiki 頁面類型

| 頁面類型 | 用途 | 範例 |
|----------|------|------|
| **摘要頁** | 單一來源的重點摘錄 | `summary-paper-x.md` |
| **實體頁** | 特定人物/組織/概念的持久頁面 | `entity-openai.md` |
| **概念頁** | 跨來源的主題彙整 | `concept-attention-mechanism.md` |
| **比較頁** | 兩個以上項目的結構化對比 | `compare-gpt4-vs-claude.md` |
| **綜合頁** | 高層次的洞察與趨勢 | `synthesis-llm-trends-2026.md` |
| **總覽頁** | wiki 的整體摘要 | `overview.md` |

### index.md — 內容導向索引

**用途**：wiki 的分類目錄，LLM 查詢時的第一站。

**結構**：
```markdown
## Entities
- [OpenAI](entity-openai.md) — AI research lab, creator of GPT series
- [Anthropic](entity-anthropic.md) — AI safety company, creator of Claude

## Concepts
- [Attention Mechanism](concept-attention.md) — Core transformer building block
- [RLHF](concept-rlhf.md) — Alignment technique using human feedback

## Sources
- [Paper: Attention Is All You Need](summary-attention-paper.md) — 2017, foundational transformer paper
```

**設計要點**：
- 按類別組織（實體、概念、來源等）
- 每頁一行：連結 + 單行摘要 + 可選元資料
- ~100 來源 / 數百頁面時仍有效，無需向量搜尋
- 每次 Ingest 時同步更新

### log.md — 時間導向日誌

**用途**：wiki 的演化時間線，記錄所有操作。

**結構**：
```markdown
## [2026-04-01] ingest | Attention Is All You Need
- Created summary page
- Updated entity-google.md, concept-attention.md
- Added 3 cross-references

## [2026-04-02] query | Compare GPT-4 and Claude 3
- Generated compare-gpt4-vs-claude.md
- Filed as permanent wiki page

## [2026-04-03] lint | Weekly health check
- Found 2 contradictions, 1 orphan page
- Fixed entity-openai.md outdated claim
```

**設計要點**：
- Append-only，永不回溯修改
- 前綴格式 `## [YYYY-MM-DD] {type} | {title}` 可被 Unix 工具解析
  - 例：`grep "^## \[" log.md | tail -5` 查看最近 5 個操作
- 幫助 LLM 理解最近做了什麼，避免重複工作

### 雙元索引的分工

```
index.md                          log.md
┌───────────────────┐             ┌───────────────────┐
│  空間維度          │             │  時間維度          │
│  "wiki 裡有什麼"   │             │  "wiki 怎麼變的"   │
│  按類別分組        │             │  按時間排列        │
│  查詢時的起點      │             │  追蹤演化歷史      │
└───────────────────┘             └───────────────────┘
```

## 第四章：核心流程 — Ingest

### Ingest 是 LLM Wiki 的核心操作

Ingest 不只是「存檔」，而是將新來源**編譯**進既有知識結構。單一來源可能觸發 10-15 個 wiki 頁面的更新。

### 完整 Ingest 流程

```
Step 1          Step 2          Step 3          Step 4
使用者加入      LLM 讀取         討論重點         產生/更新頁面
新來源          全文              與使用者          ┌─ 新增摘要頁
    │               │               │             ├─ 更新實體頁 ×N
    ▼               ▼               ▼             ├─ 更新概念頁 ×N
 raw/           LLM 提取         人引導           ├─ 標記矛盾
 article.md     關鍵資訊         強調重點          ├─ 新增交叉引用
                                                  └─ 更新總覽頁

                                Step 5          Step 6
                                更新 index       追加 log
                                    │               │
                                    ▼               ▼
                                index.md        log.md
                                新增/更新條目    append 記錄
```

### 每個步驟詳解

**Step 1 — 加入來源**：使用者將文件放入 raw sources 目錄。可用 Obsidian Web Clipper 從瀏覽器直接擷取網頁文章為 markdown。

**Step 2 — LLM 讀取**：LLM 閱讀全文，提取關鍵資訊、主張、數據。

**Step 3 — 討論**：使用者與 LLM 討論重點，引導強調什麼、忽略什麼。這一步是可選的但推薦的——使用者的判斷讓 wiki 更有價值。

**Step 4 — 級聯更新**（最關鍵的步驟）：
- 建立該來源的摘要頁
- 掃描既有 wiki，找到所有相關頁面
- 更新實體頁（新資訊、修正舊資訊）
- 更新概念頁（加入新觀點或證據）
- 標記任何矛盾（新來源 vs 舊來源）
- 加入交叉引用連結

**Step 5 — 更新索引**：在 index.md 加入新頁面條目，更新已修改頁面的摘要。

**Step 6 — 記錄日誌**：在 log.md 追加一條記錄，列出所有被觸及的頁面。

### 單筆 vs 批量 Ingest

| 模式 | 優點 | 缺點 |
|------|------|------|
| **逐一 Ingest**（推薦） | 使用者全程參與，品質高 | 較慢 |
| **批量 Ingest** | 快速處理大量來源 | 品質依賴 Schema 約束 |

## 第五章：進階流程 — Query 與 Lint

### Query 流程

Query 不只是問答，而是一個可能產生新 wiki 頁面的知識建構操作。

```
使用者提問
    │
    ▼
LLM 讀取 index.md ──> 確定相關頁面範圍
    │
    ▼
讀取相關 wiki 頁面（可能 3-10 頁）
    │
    ▼
綜合答案（附引用來源頁面）
    │
    ▼
┌─────────────────────────────────┐
│ 答案是否值得保留？               │
│                                 │
│ YES ──> 存為新 wiki 頁面        │
│         (比較表、分析、連結)      │
│         更新 index + log         │
│                                 │
│ NO ──> 留在聊天紀錄             │
└─────────────────────────────────┘
```

**知識複利**：這是 LLM Wiki 的精妙之處。一次比較分析、一個深度問答、一個發現的連結——這些不該消失在聊天歷史中。回填到 wiki 後，它們成為未來查詢可引用的持久知識。

**答案形式多樣**：
- Markdown 頁面（最常見）
- 比較表
- 簡報（Marp 格式）
- 圖表（matplotlib）
- Canvas

### Lint 流程

Lint 是 wiki 的健康檢查，定期執行以維護品質。

```
LLM 掃描全 wiki
    │
    ├── 矛盾偵測：新來源推翻舊主張？
    ├── 過時聲明：已被更新資料取代？
    ├── 孤兒頁面：無入站連結的頁面？
    ├── 缺失頁面：被提及但無專頁的概念？
    ├── 缺失引用：應有交叉連結但沒有？
    └── 資料缺口：可用網路搜尋補足的空白？
         │
         ▼
    修復 + 建議新來源/新問題
```

**Lint 的價值**：LLM 不只修復問題，還**主動建議**下一步該研究什麼、該找什麼來源。這讓 wiki 的成長有方向性。

## 第六章：關鍵設計決策與原理

### 為什麼不用 RAG？

| 維度 | RAG | LLM Wiki |
|------|-----|----------|
| 知識狀態 | 每次從零開始 | 持久累積 |
| 交叉引用 | 查詢時才拼湊 | 已預先建立 |
| 矛盾處理 | 不處理 | 已標記 |
| 綜合深度 | 受檢索品質限制 | 反映全部已讀內容 |
| 維護成本 | 無（也無累積） | LLM 負擔（接近零） |

### 為什麼 wiki 而非資料庫？

- **Markdown 是通用格式**：任何編輯器都能讀寫
- **Git 免費提供版控**：歷史、分支、協作
- **Obsidian 提供圖形視覺化**：看見知識結構
- **人類可讀**：不需要特殊工具就能瀏覽

### 為什麼 LLM 能做好這件事？

文中引用 Vannevar Bush 的 Memex（1945）——個人策展的知識庫，文件之間有關聯路徑。Bush 的願景比網路更接近 LLM Wiki：私人、主動策展、連結和文件一樣有價值。

**他無法解決的問題是：誰來做維護？**

維護知識庫的繁瑣部分不是閱讀或思考——是**記帳**：更新交叉引用、保持摘要最新、標記矛盾、跨數十頁維護一致性。人類放棄 wiki 是因為維護成本成長快於價值。LLM 不會厭倦、不會忘記更新交叉引用、一次可以觸及 15 個檔案。

### 刻意抽象的設計

文件刻意不規範：
- 具體目錄結構
- Schema 格式
- 頁面命名慣例
- 工具選擇

**原因**：每個領域、偏好、LLM 選擇都不同。正確的使用方式是將此文件交給你的 LLM agent，共同實例化適合你的版本。

```
│  │ 編輯瀏覽 │ │ Graph    │ │ Marp    │ │

└─────────────────────────────────────────┘
         │

│ • 歷史追蹤 • 分支 • 協作                │
└────────┬────────────────────────────────┘
         │
┌────────▼────────────────────────────────┐
│      LLM Agent (知識維護者)              │
│ • Claude Code / Codex / OpenCode        │
│ • 讀 Schema → 執行 Ingest/Query/Lint    │
└────────┬────────────────────────────────┘
         │
┌────────▼────────────────────────────────┐
│     可選：搜尋引擎 (qmd)                │
│ • BM25 + 向量搜尋 + LLM re-ranking     │

│ • 規模 >100 來源時需要                   │
└─────────────────────────────────────────┘
```

### 工具詳解

| 工具 | 角色 | 何時需要 |
|------|------|---------|
| **Obsidian** | 人類的 IDE——瀏覽、graph view、插件生態 | 推薦但非必要 |
| **Web Clipper** | 瀏覽器→markdown 一鍵轉換 | 網路文章為來源時 |
| **Marp** | Markdown→簡報 | 需要產出簡報時 |
| **Dataview** | 查詢 frontmatter，產生動態表格 | Wiki 頁面有 YAML frontmatter 時 |
| **qmd** | 本地搜尋引擎（BM25+向量） | Wiki 超過 ~100 來源時 |
| **Git** | 版本控制 | 永遠推薦 |

### 圖片處理流程

```
Web Clipper 擷取文章（含圖片 URL）
    │
    ▼
Obsidian 設定 → 附件資料夾 = raw/assets/
    │
    ▼
快捷鍵 Ctrl+Shift+D → 圖片下載至本地
    │
    ▼
LLM 分兩步讀取：
  1. 先讀文字內容
  2. 再分別查看引用的圖片
```

## 第八章：實踐指南 — 如何開始你的 LLM Wiki

### Step 1：選擇你的領域和工具

- 決定主題（研究、讀書、個人成長...）
- 選擇 LLM agent（Claude Code、Codex、OpenCode）
- 選擇瀏覽器（Obsidian 推薦）

### Step 2：建立基本結構

```
my-wiki/
├── raw/              # 原始來源（不可變）
│   ├── assets/       # 圖片附件
│   ├── article-1.md
│   └── paper-2.md
├── wiki/             # LLM 維護的 wiki
│   ├── index.md
│   ├── log.md
│   └── ...pages...
├── CLAUDE.md         # Schema（工作流定義）
└── .git/             # 版本控制
```

### Step 3：撰寫 Schema

將 LLM_WIKI.md 的概念分享給你的 LLM agent，請它幫你建立初版 Schema。Schema 定義：
- 頁面類型和命名慣例
- Ingest 工作流的具體步驟
- index.md 的格式
- log.md 的格式
- 任何領域特定的規則

### Step 4：開始 Ingest

從一個來源開始，逐步建立。觀察 LLM 如何處理，根據需要調整 Schema。

### Step 5：定期 Lint

每週或每 10 個來源後，請 LLM 做健康檢查。

### 關鍵心態

- **你是策展人，不是作者**——你選擇來源和方向，LLM 做記帳
- **Schema 是活文件**——隨著使用不斷調整
- **一切都是可選的**——從最小可行 wiki 開始，需要時再擴展
